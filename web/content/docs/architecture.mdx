## Overview

Caramel is designed with a clean, layered architecture. Every component has a single clear responsibility. The bot, web dashboard, and background workers all share the same PostgreSQL database and Redis instance — keeping everything in sync regardless of which surface made the change.

## Two-Layer Cache

Caramel uses a **two-layer caching system** to serve guild configuration and active moderation records at sub-millisecond speed:

1. **Redis** (hot layer) — all frequently accessed data (guild configs, active mutes, silent bans) is stored in Redis. Every read hits Redis first.
2. **PostgreSQL** (cold layer) — the source of truth. Redis is populated from Postgres on startup and kept in sync via the `CacheManager`.

The `CacheManager` is the single authority for cache writes. Every time a `GuildConfig` record is updated — whether from a slash command or from the web dashboard — it calls `CacheManager.syncGuild(guildId, updated)` immediately after the database write. This ensures Redis is never stale.

### Redis Key Patterns

```
vanity:string:{guildId}        vanity:role:{guildId}
vanity:channel:{guildId}       vanity:module:{guildId}
mod:log_channel:{guildId}      mod:muted_role:{guildId}
mod:module:{guildId}           mod:thresholds_enabled:{guildId}
mod:mute_threshold:{guildId}   mod:ban_threshold:{guildId}
silentban:{guildId}:{userId}
```

## Async Job Processing

Heavy or time-sensitive operations are offloaded to **BullMQ** workers. These run independently from the main command handler, so command responses are never delayed by background work.

| Worker               | Responsibility                                                                |
| -------------------- | ----------------------------------------------------------------------------- |
| **Vanity Worker**    | Processes `PresenceUpdate` events, assigns/revokes vanity roles               |
| **SilentBan Worker** | Deletes banned users' messages, disconnects from voice, fires ban expiry jobs |
| **Mute Worker**      | Reapplies mutes on member rejoin, fires timed mute expiry                     |

Each worker has its own dedicated Redis connection with `maxRetriesPerRequest: null` (required by BullMQ). Queue instances are defined in `src/lib/utils/` and consumed by workers in `src/workers/`.

BullMQ queue names follow the `caramel-<feature>` prefix convention (e.g. `caramel-vanity`, `caramel-silentban`).

## Sapphire Framework

Caramel is built on the [Sapphire Framework](https://sapphirejs.dev/), which provides:

- Automatic command and listener loading (file-based, no manual registration)
- Slash command and prefix command handling via a single class
- Preconditions (e.g. `GuildOnly`)
- Plugin architecture (`@sapphire/plugin-subcommands` for multi-subcommand groups)

The custom `CaramelClient` in `src/structures/` extends Sapphire's client to bootstrap Redis, attach workers to `container`, and warm the Redis cache on ready.

## Module Validation

The `src/validators/ModuleValidator.ts` file exports a `ModuleValidators` map — one async validator function per module. Each validator receives the current `GuildConfig` and the `Guild` object, checks that all required resources exist (role IDs, channel IDs), and returns a `ValidationResult` with any missing items listed.

The `/module enable` command calls the appropriate validator before flipping the module flag. This prevents modules from being activated in a broken state.

## Web Dashboard Architecture

The web portal is a **Next.js 16 App Router** application that shares the same PostgreSQL database as the bot. It connects via Prisma using `@prisma/adapter-pg` and mirrors the bot's `CacheManager.syncGuild()` call after every settings write, keeping Redis consistent regardless of which surface made the change.

Authentication is handled by **NextAuth.js v5** with Discord OAuth. The dashboard is completely protected — all pages under `/dashboard` redirect to `/login` if unauthenticated.

Key design choices:

- **Dark mode only** — enforced globally via `className="dark"` on `<html>`
- **Shared database** — no separate web DB; same schema, same data
- **Dev bypass mode** — set `NEXT_PUBLIC_DEV_BYPASS=true` to skip OAuth and use mock data for local UI development
